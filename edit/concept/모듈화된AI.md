# 도메인 모듈 기반 언어 모델 아키텍처

## 왜 필요한가

현재 LLM의 근본적인 구조적 한계는 **분기가 없다**는 것이다. Transformer 기반 모델은 입력이 들어오면 동일한 파라미터를 순서대로 통과하는 선형 구조다. 법률 문서를 읽든, 요리 레시피를 읽든, 파이썬 코드를 읽든 동일한 경로를 거친다. 도메인의 시각이 먼저 적용되는 게 아니라, 모든 것이 동시에 뒤섞인 채로 처리된다.

이 문제는 실제 사용에서 두 가지 방식으로 드러난다. Claude는 전반적으로 정확하지만, 전체 문맥을 붙들고 있으려는 성향 때문에 파기해야 할 가설이나 논의까지 컨텍스트에 남겨두어 토큰 효율이 낮다. Gemini는 반대로 최신 정보에 과도하게 가중치를 두어 초기 맥락을 잃어버리고, 대화가 길어질수록 재교육이 필요해진다. 둘 다 "도메인의 필터 없이 상식이 바로 작동"하기 때문에, 무엇이 중요하고 무엇을 버려야 하는지의 판단이 흐릿해지는 것이다.

기존 MoE(Mixture of Experts)가 이 문제를 해결하려 했지만, MoE는 확률적 혼합이다. 전문가 A의 출력 50%와 전문가 B의 출력 50%를 섞는 방식이라, 명확한 분기가 아니다. 도메인의 경계도 학습된 게이팅 네트워크가 정하기 때문에 해석이 불가능하다.

이 모델이 제안하는 것은 다르다. 임베딩 공간에서 자연스럽게 형성되는 밀도 클러스터를 기반으로 도메인 모듈을 정의하고, 도메인 모듈이 먼저 입력을 해석한 다음 그 결과를 상식 레이어로 넘기는 구조다.

---

## 왜 이 구조인가: 인간이 만든 최고의 아키텍처는 인간의 유기체를 닮는다

이 모델의 구조를 정당화하기 전에, 한 가지 철학적 관찰에서 시작한다. 인간이 만드는 것들은 무의식적으로 인간의 유기체적 구조를 따라간다. 서킷브레이커가 금융 시장과 소프트웨어 시스템에 동시에 존재하는 것, 법이 API처럼 취지-장치-흐름-입출력의 구조를 가지는 것 — 이것들은 우연이 아니다. 복잡한 문제를 다루는 시스템이 수렴하는 방향이 인간의 유기체적 설계와 유사하기 때문이다. 반대로 말하면, 인간이 만들 수 있는 최고의 아키텍처는 인간의 유기체와 가장 유사한 것이다.

그렇다면 인간의 뇌는 어떻게 작동하는가.

신경과학에서 뇌의 모듈성(modularity of mind)은 오래전부터 관찰되어온 구조다. 브로카 영역은 언어 생성을, 베르니케 영역은 언어 이해를, 방추상 얼굴 영역(fusiform face area)은 얼굴 인식을 담당한다. 각 영역이 도메인 특화된 처리를 먼저 수행한다. 그리고 그 결과들을 통합해서 최종 판단을 내리는 것이 전두엽(prefrontal cortex)의 역할이다. 전두엽은 실행 기능(executive function) — 계획, 판단, 우선순위 결정 — 을 담당하며, "지금 이 상황에서 어떤 모듈의 처리 결과를 얼마나 반영할 것인가"를 결정한다.

이 모델의 구조는 이 순서를 그대로 따른다. 도메인 모듈들이 각자의 전문 영역에서 먼저 입력을 해석하고(뇌의 특화 영역들), 라우팅 레이어가 어떤 모듈을 활성화할지 결정하며(전두엽의 실행 기능), Attention 베이스 레이어가 도메인 모듈들의 출력을 통합해서 최종 응답을 생성한다(전두엽의 통합 처리). 이것은 뇌과학에서 빌려온 비유가 아니라, 복잡계가 수렴하는 구조가 원래 그렇게 생겼다는 관찰이다.

다만 한 가지를 명확히 한다. 뇌과학은 이 구조의 직관을 설명하는 도구이고, 실제 정당화는 임베딩 공간의 기하학과 수식으로 한다. "뇌가 이렇게 작동하니까 이 모델도 이래야 한다"는 주장이 아니라, "인간의 인지와 이 모델이 같은 구조적 해답으로 수렴했다"는 관찰이다.

## 핵심 철학: 도메인이 먼저, 상식이 나중

인간의 인식 순서를 생각해보면 이 구조가 자연스럽다. 법률가는 계약서를 읽을 때 "이건 계약 위반이다"라는 도메인 해석을 먼저 하고, 그 다음에 "그러니까 상대방이 책임을 진다"는 상식적 결론으로 넘어간다. 의사도 "이 증상은 염증 반응이다"라는 전문적 판단이 먼저이고, "항염증제를 써야 한다"는 상식적 추론이 뒤따른다. 도메인 지식이 먼저 세상을 특정한 방식으로 보게 만들고, 그 위에서 상식이 작동한다.

이 모델은 이 순서를 구조적으로 구현한다.

$$\text{Input} \rightarrow \text{Routing Layer} \rightarrow \text{Domain Modules} \rightarrow \text{Base Layer (Attention)} \rightarrow \text{Output}$$

## 모듈의 경계: 칼로 나누지 않는다

모듈의 경계를 인위적으로 정하지 않는다. 의사라고 의술만 알 필요가 없고, 변호사라고 법만 알 필요가 없듯이, 도메인 모듈들 사이의 중복을 허용한다. "의료 과실"이라는 개념은 법률과 의학 두 도메인에 동시에 속한다. 이 중간 영역을 어느 한쪽에 억지로 할당하는 게 아니라, 두 모듈이 자연스럽게 공유하는 영역으로 둔다.

모듈의 경계는 임베딩 공간의 밀도 클러스터에서 귀납적으로 도출된다. 법률 관련 토큰들을 임베딩으로 찍으면 특정 영역에 밀집되어 있고, 의학 관련 토큰들은 또 다른 영역에 밀집된다. 각 클러스터의 밀도 중심 $c_m$에서 시작해서, 코사인 유사도 기반으로 관련 토큰들을 점진적으로 포함시키며 모듈의 범위를 증식한다. 두 클러스터의 밀도 중심에서 각각 증식하다 보면, 중간 영역에서 자연스럽게 겹치는 부분이 생기고 그게 공유 영역이 된다.

---

## 학습 데이터 구성

베이스 모델에서 사후에 모듈을 분리하는 방식은 이미 뒤섞인 파라미터를 다시 나누려는 시도라서 경계가 흐릿해진다. 이 모델은 처음부터 도메인별로 분리된 데이터로 각 모듈을 독립적으로 학습시킨다.

법률 모듈은 판례, 법령, 법학 논문, 계약서로 학습한다. 의학 모듈은 임상 논문, 의학 교과서, 진료 기록 형식의 텍스트로 학습한다. 코딩 모듈은 GitHub 코드, 기술 문서, Stack Overflow로 학습한다. 도메인 경계에 걸치는 텍스트들 — "의료 과실 소송"처럼 두 도메인 이상이 겹치는 경우 — 은 관련 모듈들이 공통으로 학습하여 공유 영역을 자연스럽게 형성한다.

Attention 베이스 레이어는 기존 방식 그대로 혼합 데이터로 학습한다. 잘 되던 것을 굳이 바꿀 필요가 없다.

---

## 동적 모듈 활성화: 몇 개를 불러올 것인가

고정된 숫자를 정하지 않는다. 입력의 성격이 몇 개의 모듈을 불러올지를 자연스럽게 결정한다.

입력 토큰들의 임베딩 $e_{input}$과 각 도메인 모듈의 밀도 중심 $c_m$ 사이의 코사인 유사도를 계산하고, 유사도가 동적 Threshold $\lambda$ 이상인 모듈만 활성화한다.

$$M_{active} = \{m \mid \cos(e_{input}, c_m) \geq \lambda\}$$

단순한 법률 질문은 법률 모듈 하나만 활성화되고, "의료 과실 소송"처럼 도메인 경계에 걸치는 입력은 법률 모듈과 의학 모듈이 동시에 활성화된다. 계산 비용이 입력의 복잡도에 따라 자동으로 조절된다.

$M_{active}$가 공집합이 되는 엣지 케이스 — 어떤 도메인과도 유사도가 낮은 입력 — 에는 유사도가 가장 높은 모듈 하나를 무조건 활성화하는 폴백 로직이 작동한다. 브랜치 메모리 Retriever의 Backoff 개념과 동일한 방식이다.

---

## 여러 모듈의 결과를 합치는 방법: 양방향 RNN에서 배운다

여러 모듈이 동시에 활성화될 때 그 결과를 단순히 더하거나 평균내면 MoE의 확률적 혼합과 다를 바가 없다. 양방향 RNN이 두 방향의 hidden state를 섞지 않고 concatenate하는 것처럼, 각 모듈의 출력을 독립적으로 유지하면서 concat한다.

$$O_{domain} = \text{concat}(s_m \cdot O_m \text{ for } m \in M_{active})$$

여기서 $s_m = \cos(e_{input}, c_m)$은 각 모듈의 활성화 가중치다. Weighted sum이 아닌 weighted concat을 쓰는 이유는, 법률 모듈의 관점과 의학 모듈의 관점을 섞어버리는 게 아니라 두 관점을 독립적으로 유지하면서 Attention 베이스 레이어가 "이 두 관점을 어떻게 통합할 것인가"를 학습하게 만들기 위해서다.

---

## 브랜치 메모리 Retriever와의 연결

이 모델은 브랜치 메모리 Retriever와 같은 라우팅 레이어를 공유한다. 대화가 새로운 국면에 접어드는 순간, 즉 브랜치가 분기되는 순간이 도메인 모듈이 재평가되는 순간이기도 하다. 법률 상담에서 코딩 질문으로 넘어갈 때 브랜치가 분기되는 동시에 활성화된 도메인 모듈도 교체된다. 두 시스템이 코사인 유사도 기반의 동적 Threshold라는 동일한 철학을 공유하며 자연스럽게 연동된다.

---

## 현재 열려있는 문제들

$\lambda$ 값을 어떻게 자동으로 조정할 것인가가 핵심 미해결 문제다. 법률 상담처럼 도메인이 명확한 대화와 일상 대화처럼 도메인이 모호한 경우에 적절한 $\lambda$ 값이 다를 수 있다. 또한 모듈의 밀도 중심 $c_m$을 초기에 어떻게 잡을 것인지, 새로운 도메인이 추가될 때 기존 모듈들의 밀도 중심이 어떻게 업데이트되어야 하는지도 실험으로 검증이 필요하다.

---

## 하고 싶은 말

누구든 해당 아이디어를 써도 좋다. 왜냐하면 현재 내 업무 환경에서는 GPU도 데이터도 없기 때문이다. 다만 최소한의 양심으로 댓글이던 뭐던 남겨줬으면 한다. 아무것도 할 수 없는 내 블로그에서 썩어가느니, 당신이 더 나은 세상을 만드는 데 그냥 기여했고, 나의 아이디어가 도움이 되었다는 말을 남겨주는 것이라면 그것으로 만족한다. 다만 단순한 아이디어 도둑놈이라면 다시 한 번 내 블로그 제목을 봐라. 아 물론 지금 안 봐도 된다. 지옥에서는 보기 싫어도 보게 될거다.

## 한 줄 요약

현재 LLM은 도메인의 시각 없이 상식이 바로 작동하는 구조적 한계를 가진다. 임베딩 공간의 밀도 클러스터 기반으로 중복을 허용한 도메인 모듈을 구성하고, 도메인 모듈이 먼저 입력을 해석한 다음 Attention 베이스 레이어로 넘기는 구조를 통해, 전문성과 상식이 올바른 순서로 작동하는 모델을 만들 수 있다.
